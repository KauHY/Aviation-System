# 调用路径与文件职责总览（Aviation-System）

> 目标：把“现在系统到底怎么跑起来、哪些文件真在用、哪些是历史包袱”说清楚。

## 1. 启动调用路径（真实）

1. 启动脚本：`start.bat` / `start.sh`
2. 安装依赖：`pip install -r requirements.txt`
3. 进入目录：`cd backend`
4. 运行入口：`python main.py`
5. FastAPI 应用启动：`backend/main.py`
   - `startup_event()` 加载 JSON 数据（users/tasks/flights/blockchain 等）
   - 挂载静态资源：`/static` -> `frontend/static`
   - 挂载图片资源：`/PICTURES` -> `../PICTURES`
   - 挂载 tcg 静态资源：`/tcg/static` -> `backend/tcg/static`

---

## 2. 主系统调用链（你现在主要在跑的）

### 2.1 页面链路（前端模板）
- 浏览器访问页面（例如 `/`, `/video-system`, `/blockchain-deposit`）
- `backend/main.py` 对应 `@app.get(...)` 返回 `templates.TemplateResponse(...)`
- 模板文件来源：`frontend/*.html`
- 前端 JS 再调用后端 API（`/api/...`）

### 2.2 认证链路
- 前端登录请求：`POST /api/auth/login`
- 后端入口：`backend/main.py`
- JWT 生成与校验：`create_access_token() / verify_token()`
- 用户信息来源：
  - 主数据：`backend/users.json`
  - 同步接入 tcg 用户能力：`backend/tcg/backend/app/services/auth.py`（`verify_password/get_password_hash/authorize_user/get_user_by_address`）

### 2.3 区块链与合约链路
- API 请求进入：`backend/main.py` (`/api/blockchain/*`, `/api/contract/*`)
- 调用合约引擎：`backend/contracts/contract_engine.py`
- 执行主合约：`backend/contracts/maintenance_record_master_contract.py`
- 必要时调用子链合约：`backend/contracts/aircraft_subchain_contract.py`
- 事件与链数据：`event_system.py` + JSON 文件（`blockchain.json`, `blockchain_events.json`, `contracts.json`）

### 2.4 权限链路（RBAC）
- 权限定义和检查：`backend/permission_manager.py`
- 前端权限开关：`frontend/static/permission-manager.js`
- 实际业务接口是否真正 enforce：以 `backend/main.py` 每个 API 内部逻辑为准

## 3. TCG 子系统（现状）

### 3.1 已接入内容（被主系统真实使用）
- `sys.path` 注入：`backend/main.py` -> `./tcg/backend`
- 调用 `AuthService` 的用户相关方法（哈希、校验、授权、按地址取用户）
- 挂载 `tcg/static`

### 3.2 未完全接入内容（存在但主链路没吃满）
- `backend/tcg/backend/main.py` 的独立 API 应用（并未由主 app include_router）
- `backend/tcg/backend/app/api/*` 的记录/用户 API 路由
- `tcg_templates` 已初始化，但主程序里几乎没有显式用 `tcg_templates.TemplateResponse(...)` 渲染 tcg 模板

> 结论：你现在是“部分复用 tcg 能力”，不是“完整挂载 tcg 子系统”。

## 4. 文件职责速查（按目录）

## 根目录
- `README.md`：项目说明（和真实代码有部分偏差）
- `start.bat` / `start.sh`：本地开发启动脚本
- `docker-compose.yml` / `Dockerfile`：容器部署
- `requirements.txt`：Python 依赖

## backend/
- `main.py`：**超级入口文件（核心屎山）**，包含页面路由、API、WebSocket、认证、区块链调用、任务等
- `permission_manager.py`：RBAC 权限与审计逻辑
- `contracts/`：智能合约模拟引擎与数据结构
  - `contract_engine.py`：交易执行、区块生成、链校验
  - `maintenance_record_master_contract.py`：主链业务（create/approve/release）
  - `aircraft_subchain_contract.py`：飞机子链记录管理
  - `event_system.py`：事件总线
  - `signature_manager.py` / `merkle_tree.py` / `state_root.py`：签名、Merkle、状态根
- `*.json`：运行时数据文件（用户、任务、航班、区块链、记录等）
- `tests/test_system.py`：测试入口（覆盖度未知）
- `check_*.py` / `generate_*.py` / `convert_*.py`：工具脚本（大多不是运行时必须）

## backend/tcg/
- `main.py`：tcg 的一体化 Web 版本（可独立跑）
- `backend/main.py`：tcg 的 API 版本（可独立跑）
- `backend/app/services/auth.py`：用户认证服务（主系统已复用）
- `backend/app/services/storage.py`：tcg 记录和用户 JSON 存储
- `backend/app/models/*.py`：tcg 数据模型
- `backend/app/api/*.py`：tcg API 路由
- `templates/` + `static/`：tcg 页面与样式
- `data/*.json`：tcg 子系统数据

## frontend/
- `*.html`：页面模板（由 FastAPI `TemplateResponse` 渲染）
- `static/script.js`：视频/WebRTC 主逻辑
- `static/contract-client.js`：区块链合约调用客户端（签名、nonce、请求封装）
- `static/permission-manager.js`：前端权限展示控制
- `static/load-header.js` + `unified-header*`：统一头部加载
- `add_*.py/remove_*.py/fix_*.py`：前端批处理脚本（非运行时）

## 5. 当前“屎山”风险点（最要命的 6 个）

1. `backend/main.py` 体量过大且职责混杂（页面、API、认证、WebSocket、区块链都在一起）。
2. 同一文件中存在重复片段/重复路由定义风险（后半段有大量近似重复代码）。
3. README 描述与真实接线不完全一致，文档可信度下降。
4. 主系统与 tcg 存在双实现（认证/记录逻辑重叠），边界不清。
5. JSON 文件既当数据库又当消息/状态存储，易出现并发写入和数据不一致。
6. 工具脚本与运行时代码混在同层目录，认知噪音高。
