# 调用路径与文件职责总览（Aviation-System）

> 目标：把“现在系统到底怎么跑起来、哪些文件真在用、哪些是历史包袱”说清楚。

## 1. 启动调用路径（真实）

1. 启动脚本：`start.bat` / `start.sh`
2. 安装依赖：`pip install -r requirements.txt`
3. 进入目录：`cd backend`
4. 运行入口：`python main.py`
5. FastAPI 应用启动：`backend/main.py`
  - `startup_event()` 调用 `app_state` 加载 JSON 数据（users/tasks/flights/blockchain 等）
  - 挂载静态资源：`/static` -> `frontend/static`
  - 路由拆分：`routes/pages.py` / `routes/auth.py` / `routes/blockchain.py` / `routes/system.py` / `routes/flights.py` / `routes/tasks.py` / `routes/inspection.py` / `routes/permissions.py`
  - 备注：`/PICTURES` 当前未在入口挂载，如需要可在 `main.py` 里补上

---

## 2. 主系统调用链（你现在主要在跑的）

### 2.1 页面链路（前端模板）
- 浏览器访问页面（例如 `/`, `/video-system`, `/blockchain-deposit`）
- `backend/routes/pages.py` 返回 `templates.TemplateResponse(...)`（模板实例在 `backend/templates.py`）
- 模板文件来源：`frontend/*.html`
- 前端 JS 再调用后端 API（`/api/...`）

### 2.2 认证链路
- 前端登录请求：`POST /api/auth/login`
- 后端入口：`backend/routes/auth.py`
- JWT 生成与校验：`app_state.create_access_token()` / `app_state.verify_token()`
- 服务层：`backend/services/auth_workflow.py` / `backend/services/users.py`
- 用户信息来源：`backend/users.json`

### 2.3 区块链与合约链路
- API 请求进入：`backend/routes/blockchain.py` (`/api/blockchain/*`, `/api/contract/*`)
- 服务层：`backend/services/blockchain_workflow.py`
- 调用合约引擎：`backend/contracts/contract_engine.py`
- 执行主合约：`backend/contracts/maintenance_record_master_contract.py`
- 必要时调用子链合约：`backend/contracts/aircraft_subchain_contract.py`
- 事件与链数据：`event_system.py` + JSON 文件（`blockchain.json`, `blockchain_events.json`, `contracts.json`）

### 2.4 权限链路（RBAC）
- 权限定义和检查：`backend/permission_manager.py`
- 权限查询接口：`backend/routes/permissions.py`
- 前端权限开关：`frontend/static/permission-manager.js`
- 实际业务接口是否真正 enforce：以对应路由模块内逻辑为准

## 4. 文件职责速查（按目录）

## 根目录
- `README.md`：项目说明（和真实代码有部分偏差）
- `start.bat` / `start.sh`：本地开发启动脚本
- `docker-compose.yml` / `Dockerfile`：容器部署
- `requirements.txt`：Python 依赖

## backend/
- `main.py`：入口引导（app 初始化、middleware、mount、include_router）
- `app_state.py`：全局状态与核心工具函数（JSON 读写、区块链初始化、报表数据生成）
- `templates.py`：模板实例初始化（前端模板与 tcg 模板兜底）
- `services/`：领域服务与工作流（JSON 读写、任务/认证/区块链业务组装）
- `templates.py`：模板实例初始化（前端模板）
- `routes/`：按领域拆分的路由
  - `pages.py`：页面渲染路由
  - `auth.py`：认证与用户资料接口
  - `blockchain.py`：区块链与合约接口
  - `system.py`：系统管理与报表接口
  - `flights.py`：航班列表/创建/更新
  - `tasks.py`：检测人员与任务接口
  - `inspection.py`：图片检测接口
  - `permissions.py`：权限查询接口
- `permission_manager.py`：RBAC 权限与审计逻辑
- `contracts/`：智能合约模拟引擎与数据结构
  - `contract_engine.py`：交易执行、区块生成、链校验
  - `maintenance_record_master_contract.py`：主链业务（create/approve/release）
  - `aircraft_subchain_contract.py`：飞机子链记录管理
  - `event_system.py`：事件总线
  - `signature_manager.py` / `merkle_tree.py` / `state_root.py`：签名、Merkle、状态根
- `*.json`：运行时数据文件（用户、任务、航班、区块链、记录等）
- `tests/test_system.py`：测试入口（覆盖度未知）
- `check_*.py` / `generate_*.py` / `convert_*.py`：已移动到 `tools/`（大多不是运行时必须）

## backend/tcg/
- 已合并并删除（历史目录，不再使用）

## frontend/
- `*.html`：页面模板（由 FastAPI `TemplateResponse` 渲染）
- `static/script.js`：视频/WebRTC 主逻辑
- `static/contract-client.js`：区块链合约调用客户端（签名、nonce、请求封装）
- `static/permission-manager.js`：前端权限展示控制
- `static/load-header.js` + `unified-header*`：统一头部加载
- `add_*.py/remove_*.py/fix_*.py`：前端批处理脚本（非运行时）

## 5. 当前“屎山”风险点（最要命的 6 个）

1. `backend/main.py` 体量过大且职责混杂（已拆分，入口已瘦身）。
2. 同一文件中存在重复片段/重复路由定义风险（已清理重复路由块）。
3. README 描述与真实接线不完全一致，文档可信度下降。
4. JSON 文件既当数据库又当消息/状态存储，易出现并发写入和数据不一致。
5. 工具脚本与运行时代码混在同层目录，认知噪音高。
